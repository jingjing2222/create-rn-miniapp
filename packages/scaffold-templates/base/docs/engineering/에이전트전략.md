# AI 하네스 엔지니어링 기반 운영 전략

> 목적: 에이전트가 flat monorepo 구조를 오해하지 않고, 문서, 테스트, 검증 루프를 안정적으로 반복하도록 하는 운영 기준.

## 0) TL;DR
- source of truth는 `docs/`다.
- 작업 전후로 `docs/ai/*`를 먼저, 마지막에 갱신한다.
- 현재 레포는 `frontend`, `backoffice`, `server` 중심의 단일 제품 모노레포다.
- MiniApp은 AppInToss + Granite + TDS 기준으로 운영한다.
- 루트 검증은 `{{verifyCommand}}`가 단일 진입점이다.

## 1) 현재 레포 기준선

### 1.1 디렉터리 구조
- `frontend`: Granite 기반 MiniApp 클라이언트
- `backoffice`: optional Vite 기반 운영 도구
- `server`: optional Supabase workspace
- `docs`: 제품, 엔지니어링, AI 하네스 문서

### 1.2 생성 원칙
- `frontend`는 AppInToss React Native tutorial 순서대로 생성한다.
- `server`는 `supabase init` 기준으로 생성한다.
- `backoffice`는 `create-vite --template react-ts` 기준으로 생성한다.
- 생성 이후 루트 monorepo 설정과 하네스 문서를 정리한다.

### 1.3 툴체인
- 패키지 매니저: `{{packageManagerCommand}}`
- task runner: Nx
- formatter and linter: Biome
- MiniApp runtime: AppInToss + Granite + TDS
- optional server runtime: Supabase CLI

## 2) 에이전트 작업 규칙

### 2.1 우선순위
1. `docs/ai/Plan.md`에 범위와 DoD 확인 또는 갱신
2. 테스트 또는 재현 절차 작성
3. 코드 변경
4. 검증 실행
5. `docs/ai/Status.md`를 최신 상태로 재작성
6. 브랜치 생성, 커밋, 브랜치 푸시, PR 생성

### 2.2 문서 우선 원칙
- 구현보다 문서를 먼저 맞춘다.
- 경로나 구조가 바뀌면 `AGENTS.md`와 `docs/index.md`를 즉시 동기화한다.
- 존재하지 않는 경로를 예시로 남기지 않는다.
- Granite, AppInToss, TDS 구현 판단은 공식 문서를 우선 근거로 삼는다.

### 2.3 TDD 원칙
- 버그 수정과 로직 변경은 가능한 한 실패 테스트부터 시작한다.
- 테스트가 어려우면 최소한 재현 절차와 기대 결과를 남긴다.
- 구현은 red, green, refactor 순서를 기본으로 한다.

## 3) 검증 기준
- 기본 게이트: `{{verifyCommand}}`
- reduced verification 허용 조건:
  1. docs-only 변경
  2. 코드 경로 diff 없음
- reduced verification을 택하면 사유를 `docs/ai/Status.md`에 기록한다.

## 4) 브랜치와 커밋 정책
- 기본 접두사: `feat:`, `fix:`, `docs:`, `chore:`
- 한 커밋은 한 의도만 가진다.
- 기본 정책: 기능 브랜치에서 작업 후 PR로 병합한다.
- `main` 직접 푸시는 금지한다.
- PR 병합 전 최소 품질게이트와 리뷰 통과를 필수로 한다.

## 5) 향후 확장 시 업데이트 포인트
- `server`와 `backoffice`가 optional이므로, 프로젝트별 검증 차이를 문서화한다.
- CI에 `verify` 스크립트를 고정한다.
- 문서 구조 변경 시 `docs/index.md`를 단일 진입점으로 유지한다.
