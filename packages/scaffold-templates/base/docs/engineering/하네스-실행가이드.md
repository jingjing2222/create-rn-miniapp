# 하네스 실행 가이드

## 목표
- 에이전트가 길을 잃지 않게 파일 중심으로 작업한다.
- 공식 문서와 템플릿 문서를 같이 근거로 사용한다.
- 품질게이트를 자동화해 구현, 검증, 문서 반영 루프를 반복한다.

## 디렉터리 규칙
- `docs/ai/Prompt.md`: 상위 목표와 불변 제약
- `docs/ai/Plan.md`: 범위, 단계, DoD
- `docs/ai/Implement.md`: 구현 루프와 실행 규칙
- `docs/ai/Status.md`: 최신 상태 1페이지
- `docs/ai/Decisions.md`: 중요한 결정

## 작업 루프
1. 하네스 진입: `Plan` → `Status` → `Implement` 순으로 확인한다.
2. MiniApp 작업이면 `docs/engineering/appsintoss-granite-api-index.md`에서 API 후보를 먼저 찾는다.
3. 더 넓은 카탈로그나 실무 메모가 필요하면 `docs/engineering/appsintoss-granite-full-api-index.md`를 본다.
4. 라우팅이나 페이지 구조를 건드리면 `docs/engineering/granite-ssot.md`를 먼저 확인한다.
5. UI 구현이 포함되면 `docs/engineering/tds-react-native-index.md`와 원문 TDS 문서를 먼저 확인한다.
6. TDD 기준선을 만든다.
   - 실패 테스트를 먼저 작성하거나
   - 최소한 재현 절차와 기대 결과를 기록한다.
7. 구현한다.
8. `{{verifyCommand}}`를 실행한다.
9. 실패를 수정한다.
10. 문서를 반영한다.
11. 브랜치 생성, 커밋, 브랜치 푸시, PR 생성 순으로 마무리한다.

## PR 체크리스트
- [ ] PR 본문에 실행한 검증 명령과 결과 기록
- [ ] `{{verifyCommand}}` 통과 확인
- [ ] 문서 링크와 실제 구조 정합성 확인

## 최소 품질게이트
- format 통과
- lint 통과
- typecheck 통과
- test 통과

## 브랜치 전략
- 기본: 기능 브랜치에서 작업 후 PR 생성
- `main` 직접 푸시는 금지
- PR은 `{{verifyCommand}}`와 리뷰를 통과한 뒤 병합

## 커밋 규칙
- feat, fix, docs, chore 접두사 사용
- 커밋 메시지에 목적과 영향 범위를 적는다
