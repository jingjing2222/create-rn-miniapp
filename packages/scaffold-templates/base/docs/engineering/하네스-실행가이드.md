# 하네스 실행 가이드

## 목표
- 에이전트가 길을 잃지 않게 파일 중심으로 작업한다.
- 공식 문서와 템플릿 문서를 같이 근거로 사용한다.
- 품질게이트를 자동화해 구현, 검증, 문서 반영 루프를 반복한다.

## 디렉터리 규칙
- `docs/ai/Prompt.md`: 상위 목표와 불변 제약
- `docs/ai/Plan.md`: 범위, 단계, 수정 파일, 검증 계획, DoD
- `docs/ai/Status.md`: 최신 상태 1페이지
- `docs/ai/Decisions.md`: 중요한 결정

## Frontend 기능 구현 전 체크
1. 기능 목표와 제외 범위를 `Plan`에 먼저 적는다.
2. `Plan`에 `## MiniApp 참고 문서` 섹션을 직접 추가한다.
   - framework 문서:
   - TDS 문서:
   - 권한/로딩/에러/analytics 체크:
3. `docs/engineering/appsintoss-granite-api-index.md`에서 필요한 기능 축이 존재하는지 찾는다.
4. `docs/engineering/appsintoss-granite-full-api-index.md`에서 정확한 API, 타입, 에러, 공식 URL을 확인한다.
5. 라우팅, 페이지 파일, params를 건드리면 `docs/engineering/granite-ssot.md`를 먼저 확인한다.
6. UI 구현이면 `docs/engineering/tds-react-native-index.md`와 원문 TDS 문서를 먼저 확인한다.
7. 위에서 확인한 문서를 `Plan`의 `MiniApp 참고 문서` 섹션에 적는다.
8. 권한, 로딩, 에러, analytics가 필요한 기능인지 구현 전에 체크한다.
9. 수정할 파일과 구현 순서를 `Plan`에 적는다.
10. 실패 테스트 또는 재현 절차를 먼저 만든다.

## 작업 루프
1. 하네스 진입: `Plan` → `Status` → `Decisions` 순으로 확인한다.
2. frontend 기능 구현이면 위 `Frontend 기능 구현 전 체크`를 먼저 따르고, `Plan`의 `MiniApp 참고 문서` 섹션을 채우기 전에는 구현을 시작하지 않는다.
3. MiniApp 작업이면 `docs/engineering/appsintoss-granite-api-index.md`에서 기능 축을 먼저 찾는다.
4. 정확한 URL, 세부 타입, 실무 메모가 필요하면 `docs/engineering/appsintoss-granite-full-api-index.md`를 본다.
5. 라우팅이나 페이지 구조를 건드리면 `docs/engineering/granite-ssot.md`를 먼저 확인한다.
6. UI 구현이 포함되면 `docs/engineering/tds-react-native-index.md`와 원문 TDS 문서를 먼저 확인한다.
7. TDD 기준선을 만든다.
   - 실패 테스트를 먼저 작성하거나
   - 최소한 재현 절차와 기대 결과를 기록한다.
8. 구현에 들어가기 전 `Plan`에 수정 파일과 구현 순서를 적는다.
9. 구현 경로가 바뀌면 코드를 계속 쓰기 전에 `Plan`을 먼저 갱신한다.
10. 구현한다.
11. `{{verifyCommand}}`를 실행한다.
12. 실패를 수정한다.
13. 문서를 반영한다.
14. 브랜치 생성, 커밋, 브랜치 푸시, PR 생성 순으로 마무리한다.
